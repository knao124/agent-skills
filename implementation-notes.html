Implementation Notes

Updated: 2026-05-20 18:11:29 JST

Summary

Implement a new tweet explanation skill that handles X/Twitter post URLs by trying browser-based public extraction first, falling back to the local App-only token at ~/.x-token, and then using the existing explain-to-html skill to produce the final explanation article.

Final implementation adds skills/tweet-explainer/SKILL.md, a token-backed fallback helper at skills/tweet-explainer/scripts/fetch_x_post.py, trigger/output eval metadata, and README discovery/install entries.

Design Decisions

  • Status: decided. Context: token usage should be avoided when browser extraction can collect enough content. Note: The skill body will define a browser-first decision gate and a deterministic API fallback script. Rationale: Browser availability depends on the active Codex environment, while token fallback should be repeatable and low ambiguity. Impact: Token usage is reserved for cases where the browser path cannot reliably collect the post/article data.
  • Status: decided. Context: App-only token must come from ~/.x-token. Note: The fallback script will read that file directly, strip an optional Bearer prefix, and never echo the token. Rationale: This matches the requested local setup and avoids asking agents to copy secrets into commands. Impact: Missing token file becomes a clear stop condition with a user-facing error.

Deviations

None.

Tradeoffs

  • Status: decided. Context: Implement browser extraction as reusable code or as skill workflow. Note: Keep browser extraction in the skill workflow and implement only the token-backed API path as a script. Rationale: Browser tools vary between Codex contexts and may require logged-in UI state, while the API path benefits from deterministic URL parsing, field selection, and normalized JSON output. Impact: The skill remains portable and still gives agents explicit browser-first instructions.
