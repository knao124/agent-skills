# Reading Note

この skill は、Codexが本やPDFを代理で精読し、読者がノートだけを読んでも、精読後に得るべき理解結果へ到達できる「精読代替ノート」を作る。

目的は、短い要約や読書の入口を作ることではない。初見の読者が、何を理解すべきか、著者がどの順番で論理を組み立てているか、登場概念がなぜ必要か、主張がどの根拠と前提で成り立つかを追えるようにする。

ここでいう「精読と同等」は、本文の全表現を記憶することではない。読者目的に照らして、主題、論理構造、主要主張、概念の役割、重要な手順や公式、著者の前提、適用限界、反例、理解確認の問いを再構成できる状態を指す。読者目的が未指定なら「初見者が説明、適用、批判までできる」を既定値にする。

## 必須ルール

- 作業順と出力順を分ける。作業中の成果は仮説として扱い、本文を読むたびに更新する。
- 「精読後に残るべき成果」は最初に仮定義し、最後に到達監査を通して最終版としてノート冒頭に出す。
- 圧縮された論理の流れから始めない。全体地図、単純例、問いの連鎖で初見者の足場を作ってから圧縮ロジックを置く。
- 用語を単なる一覧や辞書にしない。各語について、なぜ必要か、論理上どこで効くか、誤解すると何が壊れるかを書く。
- 重要概念だけでなく、重要主張を必ず台帳化する。理解の中心は「語」ではなく「著者が何をどう主張しているか」である。
- 本文由来の例、図表、公式と、Codexが作る理解補助例を明確に区別する。
- ソース内容を捏造しない。本文、PDF、URL、書名などの根拠が読めない場合は、その制約を明示する。
- 長い引用を避ける。必要な短い引用以外は、要約とパラフレーズで書く。
- 推測は推測として扱う。目次や本文からの推定、抽出失敗、根拠不足、信頼度の低い箇所を明示する。
- ノートで代替できない残差を隠さない。原文の語り口、詳細事例、演習による身体化、専門判断が残る場合は「本に戻るべき箇所」に明記する。
- 金融、法律、医療、税務などの高リスク領域では、学習用の理解整理として書き、専門助言として扱わない。

## 作業手順

1. ソース状態と読者目的を確認する。
   - ローカルパス、PDF、URL、本文が与えられている場合は、まず中身を確認する。
   - PDFでは、抽出できた範囲、目次、章見出し、図表、公式、事例、まとめ部分、OCRや抽出失敗を記録する。
   - 読者目的が未指定なら「初見者が説明、適用、批判までできる」を既定値にする。
   - 書名だけで本文にアクセスできない場合は、内容を作らず、ファイル、URL、本文提示を求める。

2. この本が作る対象を特定する。
   - 「この本は読者に何を作れる、判断できる、説明できるようにする本か」を問う。
   - テーマ名だけでなく、成果物、判断手順、説明モデル、批判軸として表現する。
   - 例: 企業価値評価モデル、意思決定手順、因果説明、歴史解釈、運用フレーム。

3. 精読後に残るべき成果を仮定義する。
   - 読者目的と本が作る対象に照らし、ノートで再現すべき理解結果を先に仮説化する。
   - 成果ごとに、どの章、主張、概念、手順、自己テストで検証するかを予定する。
   - この仮説は固定しない。章を読むたびに足し、削り、言い換える。

4. 章または主要セクションごとに差分を潰しながら読む。
   - 各章について、答える問い、本章の答え、次の問い、重要主張、必要概念、本文例、図表、公式、未解決差分を記録する。
   - 重要主張は、主張、根拠または本文位置、依存概念、前提、反論・限界、この主張でできる判断に分解する。
   - 重要概念は、平たく言うと、なぜ必要か、論理上の位置、誤解すると何が壊れるか、できるようになることに分解する。
   - 章を読むたびに、精読後成果の仮説と未解決差分を更新する。

5. 初見者向けの導入層を作る。
   - 最小限の全体地図を作る。技術書や実務書では、入力、処理、出力、検証方法のような構造にする。
   - 単純例を1つ置く。本文由来ならその旨を明示し、Codex作成例なら「理解補助例」と明示する。
   - 議論を「問い -> 著者の答え -> その答えが生む次の問い」に変換する。

6. 精読結果を再構成する台帳を作る。
   - 章ごとの役割を、内容要約ではなく全体ロジック上の機能として書く。
   - 重要主張台帳と重要概念台帳を作る。
   - 必要に応じて、手順、公式、判断分岐、ケース適用、反例、限界をジャンル別補助セクションにまとめる。
   - 圧縮版の論理の流れは、導入層、章役割、主張台帳の後に置き、復習用の高密度な復元として扱う。

7. 理解ゴールと自己テストを作る。
   - 理解ゴールは「XXをYYできる」に近い検証可能な形にする。
   - 説明、構造復元、用語運用、計算または手順適用、比較、診断、批判を必要に応じて含める。
   - 自己テストには、模範解答または採点基準を必ず付ける。

8. 最終差分監査を行い、未達ならノート本文を修正する。
   - 成果ごとに、ノート内で学べる場所、根拠、自己テスト、残る差分、本に戻るべき箇所を照合する。
   - 差分カテゴリは、論証再現、実務再現、判断基準、誤解潰し、根拠追跡、演習・身体化を基本にする。
   - 監査で重要な未達が出た場合は、監査表に書くだけで終わらせず、該当する導入、台帳、理解ゴール、自己テストへ戻って修正する。
   - ノートでは代替不能な残差だけを、理由つきで最後に残す。

## ジャンル別補助セクションの選び方

- 技術書・実務書: 手順、公式、入力値、出力値、判断分岐、失敗例。
- 投資・会計・法律・税務: ケース適用、前提、リスク、専門助言ではない注記。
- 歴史: 年表、因果連鎖、主体別利害、史料上の限界。
- 哲学・思想: 論証、反論、前提、概念対立。
- 文学: 語り、構造、主題、本文に戻るべき表現。
- 試験用途: 頻出論点、誤答パターン、採点基準。

## 文体

- 書評ではなく、学習と理解のための道具として書く。
- 見積もる、比較する、切り分ける、説明する、検証する、批判する、のような具体的な動詞を使う。
- 繰り返し構造は表にするが、表だけで分かりにくい箇所は短い文章で補う。
- 「なぜ重要か」は文章で説明し、全体を箇条書きだけにしない。
- ユーザーが案だけを求めている場合は、同じ構造を短縮版で示し、ソースに基づく最終ノートではないことを明示する。

## 完了チェック

終了前に、次を確認する。

- ソース状態、読者目的、読めた範囲、制約が明示されている。
- 精読後成果が目的依存で定義され、自己テストと対応している。
- 圧縮ロジックより前に、初見者向けの全体地図、単純例、問いの連鎖がある。
- 単純例が本文由来かCodex作成の理解補助例か区別されている。
- 章ごとの役割が、全体ロジック上の機能として説明されている。
- 重要主張台帳に、根拠、依存概念、前提・限界、この主張でできる判断がある。
- 重要概念台帳に、なぜ必要か、誤解すると何が壊れるか、できるようになることがある。
- 理解ゴールが検証可能な形になっている。
- 自己テストに模範解答または採点基準がある。
- 差分監査で未達が出た場合、ノート本文へ戻って修正している。
- それでも残る差分は、本に戻るべき箇所または代替不能な残差として明示されている。
