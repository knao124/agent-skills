# reading-note

本、PDF、記事、文書をCodexが代理で精読し、読者がノートだけを読んでも、精読後に得るべき理解結果へ到達できる「精読代替ノート」を作る skill です。

## 使う場面

- 本やPDFの内容について、ノートだけで精読後と同等の理解結果に到達したい。
- テーマ、論理の流れ、主張を支えるロジックツリー、章ごとの役割を整理したい。
- 初見者が最初に詰まる「そもそもXとは？」を先に潰したい。
- 重要主張と登場概念を、定義だけでなく「なぜ必要か」「何を判断できるか」まで理解したい。
- 「XXをYYできる」という理解ゴールに分解したい。
- 自己テストと採点基準で、理解できたかを確認したい。
- 本とノートの差分、代替不能な残差、本に戻るべき箇所を明示したい。

## 重要な考え方

この skill は、短い要約を作るものではありません。読者目的に応じて「精読後に残るべき成果」を仮定義し、本文を読むたびに更新し、最後に差分監査を通してノート本文へ戻って修正します。

精読代替の中心は、見出しを増やすことではなく、`仮説を立てる -> 本文で更新する -> 差分を監査する -> ノートを修正する` という作業ループです。

用語は単なる一覧にしません。重要概念台帳では、平たく言うと何か、なぜ必要か、論理上どこに置かれるか、誤解すると何が壊れるか、何ができるようになるかを整理します。

さらに、重要主張台帳を作ります。主張、根拠または本文位置、依存概念、前提・限界、この主張でできる判断を分けることで、本の理解を「単語」ではなく「著者の論理」の復元として扱います。

初見者向けの入口を重視します。タイトル、目次、冒頭、最上位主張に含まれる中心語について、読者が最初に聞く「そもそもXとは何か」をロジックツリー前に答えます。

全体像だけで止めません。中心主張を支えるロジックツリーを作り、主要ノードごとに、主張、親ノードへの貢献、下位ロジック、詳細解説、必要概念、本文例、誤解ポイント、次の問い、全体像へ戻る導線を入れます。

出力本文では、作業報告や構成変更理由を冒頭に書きません。タイトルの次はすぐ「この本は何を作る本か」に入り、ソース状態や注意書きは末尾の付録や監査情報に回します。

## 出力の基本形

1. この本は何を作る本か
2. 初見者が最初に詰まる問い
3. 全体地図
4. ひとつの単純例
5. 最上位主張
6. ロジックツリー全体像
7. ロジックツリーの読み方
8. 各ロジックノードの詳細解説
9. 問いの連鎖
10. 章ごとの役割
11. 重要主張台帳
12. 重要概念台帳
13. 圧縮版の論理の流れ
14. ジャンル別補助セクション
15. 到達確認
16. 自己テストと採点基準
17. 本との差分監査
18. 本に戻るべき箇所・代替不能な残差
19. 1分説明
付録. ソース状態・信頼範囲
