# Ichikura Business Plan (一倉定式 経営計画書)

この skill は、一倉定『社長学シリーズ 経営計画・資金運用』の枠組みに準拠した経営計画書を、社長(ユーザー)と共同で作成・監査・改訂する。

最重要原則: **方針書の中身(決意)は社長のものであり、agent が捏造してはならない。** 一倉の教義では、社長が自分で考えて決めたことにだけ「やり遂げる執念」が宿る。agent の仕事は、(1)枠組みと骨子の提供、(2)公開情報・資料からの材料整理、(3)数字の検算と矛盾の指摘、(4)社長の決定の文章化である。社長にしか決められない箇所は埋めずに ◆ マークで残し、明示的に質問する。

## 正式構成(これに準拠しているかで監査する)

経営計画書は「目標(仏)」と「方針書(魂)」の両方があってはじめて成立する。目標数字だけの計画は「仏つくって魂入れず」。

```text
第Ⅰ部 長期経営計画(3〜5カ年)
  目標            … 売上・経常利益(出発点)・必要なら時価総額/PBR/PER・純資産
  一、方針書      … 魂。下記6項目
  二、利益計画    … 必要利益から逆算する。願望ではなく必要額として根拠を書く
  三、販売計画    … 何を・どこで・どれだけ。事業特性に応じて翻訳する
  四、設備計画    … 売上と利益をあげる内部条件(IT/AI投資を含む)
  五、要員計画    … 内部条件その2
  六、資金運用計画 … 資金調達+目標貸借対照表。目標B/Sで計画は完成する
第Ⅱ部 短期経営計画書(当年度)
  年度方針 / 年度利益計画 / 年度の実行計画 / 投資・要員・資金
  運用: 経営計画発表会・月次チェック・年度末全面改訂
```

方針書の6項目: 基本方針 / 商品に関する方針 / 得意先に関する方針 / 販売促進に関する方針 / 未来事業に関する方針 / 内部体制整備に関する方針。

基本方針の冒頭には「顧客第一主義」(顧客が誰かの定義を含む)と「重点主義」(何をやり、何を封印するかの序列)を必ず置く。

方針書の三要件:

1. 我社の**将来**に関することだけを書く。過去の分析・反省・言い訳は書かない(「死んだ子の年を数えるな」)。必要なのは現在地の確認と「これからどうするか」だけ
2. **社長の決意**を示す。社員への要望・不満を並べるのは「業績は社員で決まる」と思っている証拠であり誤り。業績は社長の決意如何で決まるという信念で、社長自身が何をするかを書く
3. **具体的かつ簡潔**。「販売体制の強化」のようなスローガンは方針ではない。固有名詞と数字のレベルまで下ろす。ただし細部計画を盛り込みすぎると大切なことがボケる。急所を短い文章で示す

## 作業手順

1. **材料収集**: 会社のHP・IR資料・中期経営計画・紹介資料・直近決算を読む。ユーザーが資料を渡したらそれを最優先で読む。公開情報は出典を示す
2. **社長への質問**: 骨子を書く前に、社長にしか決められないことを質問する。最低限: ①目標数字(売上・経常利益。上場企業なら時価総額・マルチプルも) ②重点の序列(何をやり、何を封印するか) ③初年度に何をするか ④既存事業のランク ⑤計画書の読み手。一度に全部でなくてよいが、回答なしに本文を書かない
3. **骨子 → ドラフト**: 回答を反映して骨子、次に本文ドラフトを作る。文体は社長の一人称・断定体(「我社は〜する」)。未決定箇所は ◆ で残し、末尾に「未決事項(社長が決めて埋める)」リストを置く
4. **数字の検算**: 利益計画は必要利益から逆算し、目標の含意を隠さず明示する。検算の定石は references/framework.md を読む。緊張点(例: 目標の過半がマルチプル拡大依存、買収資金の大半が借入依存、エクイティ調達と希薄化の衝突)を見つけたら、ぼかさずに本文の「前提」として書く
5. **項目監査**: 上記「正式構成」に対して何が欠けているかを表で示す。既存の計画書を持ち込まれた場合はここから始める
6. **書き出し**: 既定は Markdown。docx 等への変換を求められたら対応する(docx skill があればそれを使う)

## 書き方の規律

- 数字はすべて検証可能な仮置きとして出し、根拠(逆算式)を本文に残す。出典のない数字を断定で書かない
- 整合性の検算をドラフトに埋め込む(例: 純資産×PBR ≒ 時価総額 ≒ 純利益×PER の二経路点検)。前提を動かすと検算が割れるようにしておくと、改訂時に矛盾を自動検出できる
- 封印事項(やらないこと)を明文化する。重点主義は「やること」より「やらないこと」で示すほうが強い
- 「必達」と「規律」が衝突する箇所(例: 必ず買収する vs 高く買わない)を見つけたら、どちらを最後に守るかを社長に決めさせる
- 完成したドラフトを渡すとき、「この文書は社長が自分の手で書き直して初めて魂が入る。これは叩き台である」と必ず伝える

## 事業特性への翻訳

製造業以外(ファンド、M&Aロールアップ、SaaS、受託等)では「商品」「得意先」「設備」が素直に当てはまらない。商品方針の本質は「重点を決める単位」の定義なので、まず単位を定義してから6項目に翻訳する。翻訳パターン集と詳細な項目別ガイドは references/framework.md を読む。
